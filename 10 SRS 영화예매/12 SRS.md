# 온라인 영화 예매 시스템 SRS

## Software Requirements Specification

## 1. 문서 개요

### 1.1 목적

본 문서는 **온라인 영화 예매 시스템**의 요구사항을 정의한다.
시스템은 사용자가 영화, 극장, 상영 일정, 좌석을 조회하고 원하는 좌석을 선택하여 결제 후 예매를 확정할 수 있도록 지원한다.

또한 MSA 설계를 전제로 하므로, 단순 기능 요구사항뿐 아니라 **서비스 분리, 데이터 정합성, 동시성, 장애 대응, 확장성**에 영향을 주는 비기능 요구사항도 함께 정의한다.

---

## 2. 시스템 개요

### 2.1 시스템 목적

온라인 영화 예매 시스템은 다음 목적을 가진다.

* 사용자가 온라인에서 영화와 상영 일정을 조회할 수 있다.
* 사용자가 실시간 좌석 현황을 확인하고 좌석을 선택할 수 있다.
* 사용자가 결제를 완료하면 예매가 확정되고 티켓이 발급된다.
* 예매 취소 및 환불 정책에 따라 취소/환불 처리가 가능하다.
* 관리자는 영화, 극장, 상영관, 상영 일정, 좌석, 예매 현황을 관리할 수 있다.
* 대량 트래픽 상황에서도 좌석 중복 예매를 방지해야 한다.

---

## 3. 분석 가정

현재 상세 정책이 모두 주어지지 않았으므로 다음을 **가정**한다.

| 구분       | 가정                                          |
| -------- | ------------------------------------------- |
| 회원 정책    | 예매는 회원만 가능하고, 비회원은 영화/상영 정보 조회만 가능하다고 가정한다. |
| 좌석 선점 시간 | 좌석 선택 후 결제 완료까지 기본 5분 동안 임시 선점된다고 가정한다.     |
| 결제 방식    | 카드 결제, 간편 결제 등 외부 PG 연동이 존재한다고 가정한다.        |
| 티켓 발급    | 결제 성공 후 모바일 티켓 또는 QR 티켓이 발급된다고 가정한다.        |
| 취소 정책    | 상영 시작 전 일정 시간까지 예매 취소가 가능하다고 가정한다.          |
| 환불 정책    | 결제 수단과 취소 시점에 따라 전액 또는 부분 환불이 가능하다고 가정한다.   |
| 관리자      | 전체 관리자와 극장 관리자 역할이 구분될 수 있다고 가정한다.          |
| 아키텍처     | MSA 기반으로 서비스별 DB 분리를 고려한다고 가정한다.            |

## 4. 사용자 유형

| 사용자 유형    | 설명                                        |
| --------- | ----------------------------------------- |
| 비회원       | 영화, 극장, 상영 일정, 좌석 일부 정보를 조회할 수 있는 사용자     |
| 일반 회원     | 로그인 후 예매, 결제, 취소, 티켓 조회가 가능한 사용자          |
| 관리자       | 영화, 극장, 상영관, 상영 일정, 좌석, 예매 정책 등을 관리하는 사용자 |
| 극장 관리자    | 특정 극장의 상영관, 좌석, 상영 일정을 관리하는 사용자           |

| 알림 수신자    | 예매 완료, 취소, 환불, 티켓 발급 알림을 받는 사용자           |

# 13. 요구사항 분석 결론

온라인 영화 예매 시스템의 핵심은 단순한 영화 조회나 예매 CRUD가 아니라, 다음 세 가지를 안정적으로 보장하는 것이다.

1. **동일 좌석 중복 예매 방지**
2. **예매-결제-좌석 상태의 정합성 유지**
3. **결제 실패, 만료, 취소, 환불에 대한 보상 처리**

따라서 후속 설계 단계에서는 다음 사항을 중점적으로 다뤄야 한다.

* 좌석 선점 저장소와 TTL 전략
* 예매/결제 Saga 흐름
* 이벤트 기반 예매 확정 및 티켓 발급
* 중복 요청 방지를 위한 Idempotency Key
* 결제 승인 이벤트 중복 처리
* Outbox Pattern 적용 여부
* 장애 발생 시 재처리 및 DLQ 전략
* 조회 트래픽과 예매 트래픽의 분리
* 서비스별 데이터 소유권과 API 경계 정의
