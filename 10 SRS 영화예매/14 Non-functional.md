아래는 위 SRS에서 **비기능적 요구사항**만 요약 정리한 내용입니다.

# 비기능적 요구사항 요약

## 1. 성능

시스템은 사용자 조회와 예매 흐름에서 빠른 응답 속도를 제공해야 한다.

| 항목          | 목표                   |
| ----------- | -------------------- |
| 영화 목록 조회    | 평균 300ms 이하          |
| 상영 일정 조회    | 평균 500ms 이하          |
| 좌석 현황 조회    | 평균 500ms 이하          |
| 좌석 선점 요청    | 평균 300ms 이하          |
| 예매 생성       | 평균 500ms 이하          |
| 결제 요청 내부 처리 | 외부 PG 응답 제외 500ms 이하 |
| 관리자 통계 조회   | 3초 이하                |

특히 **좌석 조회, 좌석 선점, 예매 생성**은 사용자 체감 속도와 직접 연결되므로 우선적으로 최적화해야 한다.

---

## 2. 확장성

시스템은 트래픽 증가에 대응할 수 있도록 서비스별 수평 확장이 가능해야 한다.

주요 확장 대상은 다음과 같다.

| 영역           | 확장 방향                 |
| ------------ | --------------------- |
| 영화 조회        | 캐시, 읽기 전용 저장소 활용      |
| 상영 일정 조회     | 캐시, 조회 모델 분리          |
| 좌석 조회        | 빠른 상태 조회 저장소 활용       |
| 예매 서비스       | 독립적인 수평 확장            |
| 결제 서비스       | 외부 PG 장애와 트래픽을 고려한 확장 |
| 알림 / 통계 / 티켓 | 비동기 메시지 기반 확장         |

특히 인기 영화 예매 오픈처럼 특정 상영 일정에 트래픽이 몰리는 상황에서도 **좌석 중복 예매가 발생하지 않아야 한다.**

---

## 3. 가용성

핵심 예매 서비스는 높은 가용성을 제공해야 한다.

| 항목         | 요구사항                                             |
| ---------- | ------------------------------------------------ |
| 핵심 서비스 가용성 | 월 99.9% 이상                                       |
| 핵심 인프라     | API Gateway, Redis, Message Broker, DB는 HA 구성 필요 |
| 외부 결제 장애   | Circuit Breaker, Timeout, Retry 적용               |
| 알림 서비스 장애  | 예매 확정 흐름에 영향 주지 않도록 격리                           |
| 통계 서비스 장애  | 예매/결제 기능에 영향 주지 않도록 격리                           |

즉, 알림이나 통계 같은 부가 기능 장애가 **예매, 결제, 좌석 선점 같은 핵심 기능을 멈추게 해서는 안 된다.**

---

## 4. 데이터 정합성

온라인 영화 예매 시스템에서 가장 중요한 비기능 요구사항은 **데이터 정합성**이다.

| 항목          | 요구사항                                |
| ----------- | ----------------------------------- |
| 좌석 중복 예매 방지 | 동일 상영 일정의 동일 좌석은 한 번만 예매 가능         |
| 좌석 선점 정합성   | 선점 사용자, 선점 시간, 예매 건이 일관되어야 함        |
| 예매-결제 정합성   | 결제 성공 후 예매 확정 누락이 장기간 지속되면 안 됨      |
| 결제-환불 정합성   | 취소/환불 상태는 추적 가능해야 함                 |
| 이벤트 중복 처리   | 동일 이벤트가 여러 번 전달되어도 결과는 한 번만 반영      |
| 최종적 정합성 허용  | 알림, 통계, 정산은 eventual consistency 허용 |
| 분산 트랜잭션 회피  | Saga 또는 보상 트랜잭션 방식 고려               |

특히 **예매, 좌석, 결제**는 강한 정합성이 필요하고, **알림, 통계, 정산**은 최종적 정합성으로 처리할 수 있다.

---

## 5. 동시성

동시에 많은 사용자가 같은 좌석을 예매하려는 상황을 안전하게 처리해야 한다.

중요 동시성 요구사항은 다음과 같다.

| 항목              | 요구사항                              |
| --------------- | --------------------------------- |
| 동일 좌석 동시 선점     | 하나의 요청만 성공해야 함                    |
| 결제 중 좌석 만료      | 결제 성공과 선점 만료가 충돌해도 최종 상태가 일관되어야 함 |
| 예매 취소와 결제 완료 충돌 | 정책에 따라 하나의 상태로 수렴해야 함             |
| 중복 API 요청       | 더블 클릭, 네트워크 재시도로 중복 예매/결제 발생 금지   |
| 관리자 좌석 차단 충돌    | 예매와 좌석 차단이 동시에 발생해도 안전해야 함        |

이 영역은 향후 설계에서 **Redis 원자 연산, DB Unique Constraint, Idempotency Key, 분산락 여부**를 반드시 검토해야 한다.

## 6. 보안

사용자 개인정보, 결제 정보, 관리자 권한을 안전하게 보호해야 한다.

| 항목       | 요구사항                      |
| -------- | ------------------------- |
| 인증       | 예매, 결제, 취소, 티켓 조회는 인증 필요  |
| 인가       | 관리자 기능은 관리자 권한 필요         |
| 개인정보 보호  | 이메일, 휴대폰 번호 등 안전하게 저장/전송  |
| 결제 정보 보호 | 카드번호 등 민감 결제 정보 직접 저장 금지  |
| 통신 보안    | HTTPS 사용                  |
| 감사 로그    | 로그인, 결제, 환불, 관리자 변경 작업 기록 |
| 토큰 관리    | 인증 토큰 만료 및 탈취 피해 최소화      |
| 입력값 검증   | 모든 API 요청값 서버 검증          |

특히 결제 정보는 내부 시스템에 직접 저장하지 않고, 외부 PG의 토큰이나 거래 ID 중심으로 관리하는 방향이 적절하다.

## 7. 장애 복구

시스템은 실패를 전제로 설계되어야 하며, 장애 발생 후에도 데이터가 복구 가능해야 한다.

| 장애 상황            | 복구 요구사항                |
| ---------------- | ---------------------- |
| 결제 실패            | 예매 실패 처리 또는 좌석 해제      |
| 결제 성공 후 예매 확정 실패 | 재처리 또는 보상 처리           |
| 이벤트 처리 실패        | 재시도 후 DLQ 저장           |
| 좌석 선점 만료 처리 실패   | 장애 복구 후 만료 데이터 정리      |
| 환불 실패            | 재시도 또는 관리자 수동 처리       |
| 부가 서비스 장애        | 핵심 예매/결제 기능으로 장애 전파 방지 |

즉, 결제 성공 후 예매 확정 실패처럼 돈과 좌석이 엮이는 상황은 반드시 **재처리 가능한 구조**여야 한다.

## 8. 관측성 / 모니터링

MSA 환경에서는 서비스 간 흐름을 추적할 수 있어야 한다.

필수 관측성 요구사항은 다음과 같다.

| 항목    | 요구사항                                       |
| ----- | ------------------------------------------ |
| 중앙 로그 | 모든 서비스 로그를 중앙에서 조회                         |
| 분산 추적 | 예매 생성 → 결제 → 예매 확정 → 티켓 발급 흐름 추적           |
| 주요 지표 | API 응답 시간, 오류율, 결제 실패율, 좌석 선점 실패율, 메시지 적체량 |
| 운영 알림 | 오류율, 지연 시간, 메시지 적체, 결제 실패율 임계치 초과 시 알림     |
| 감사 추적 | 관리자 변경, 환불 처리, 강제 좌석 변경 이력 추적              |

특히 예매 실패나 결제 불일치가 발생했을 때, 운영자가 어느 단계에서 문제가 생겼는지 빠르게 파악할 수 있어야 한다.

## 9. 운영 편의성

시스템은 안정적으로 배포, 운영, 복구할 수 있어야 한다.

| 항목        | 요구사항                       |
| --------- | -------------------------- |
| 무중단 배포    | 핵심 서비스는 가능한 한 무중단 배포 지원    |
| 설정 외부화    | 환경별 설정값 분리                 |
| 헬스 체크     | 각 서비스 상태 확인 API 제공         |
| 롤백 가능성    | 배포 실패 시 이전 버전으로 복구 가능      |
| DB 마이그레이션 | 스키마 변경 이력 관리               |
| 장애 대응 매뉴얼 | 결제, 메시지, 좌석 선점 장애 대응 절차 필요 |

MSA에서는 서비스 수가 많아지므로, 운영 자동화와 배포 안정성이 중요하다.

## 10. 호환성 / 이식성

시스템은 다양한 클라이언트와 외부 시스템 변경에 유연하게 대응해야 한다.

| 항목        | 요구사항                           |
| --------- | ------------------------------ |
| 웹/모바일 호환  | 웹과 모바일 앱에서 모두 사용 가능한 API 제공    |
| PG 교체 가능성 | 특정 결제사에 강하게 종속되지 않도록 결제 연동 추상화 |
| 알림 채널 확장성 | 이메일, SMS, 앱 푸시 등 채널 추가 가능      |

결제와 알림은 외부 서비스 의존도가 높기 때문에, 구현 시 인터페이스 기반으로 분리하는 것이 좋다.

## 결론

온라인 영화 예매 시스템의 비기능 요구사항에서 가장 중요한 축은 다음 네 가지입니다.

```text
성능
→ 좌석 조회와 선점은 빠르게 처리되어야 한다.

정합성
→ 같은 좌석이 두 번 예매되면 안 된다.

장애 복구
→ 결제 성공 후 예매 확정 실패 같은 상황을 복구할 수 있어야 한다.

확장성
→ 인기 영화 예매 오픈 시 대량 트래픽을 감당해야 한다.
```

따라서 후속 MSA 설계에서는 특히 다음 항목을 중점적으로 다뤄야 합니다.

* Redis 기반 좌석 임시 선점
* DB Unique Constraint 기반 최종 중복 방지
* Idempotency Key 기반 중복 요청 방지
* Saga 패턴 기반 예매/결제 보상 처리
* Outbox Pattern 기반 이벤트 발행 안정성
* DLQ 기반 실패 이벤트 재처리
* Circuit Breaker 기반 외부 PG 장애 격리
* 분산 추적 기반 예매/결제 흐름 관측성
