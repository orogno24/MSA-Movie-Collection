아래는 첨부된 **요구사항 분석 Agent Prompt** 기준으로, MSA 관점에서 **서비스 책임 경계**와 **의존성 최소화**를 우선해 진행한 요구사항 분석입니다. 

## 1. 시스템 개요

“영화 컬렉션 관리 시스템”은 사용자가 보유하거나 감상한 영화 정보를 등록, 조회, 수정, 삭제하고, 장르, 배우, 감독, 평점, 시청 상태, 소장 위치 등을 관리하는 시스템입니다.

MSA 관점에서는 하나의 거대한 영화 관리 서비스로 만들기보다, 변경 이유와 데이터 소유권을 기준으로 서비스를 분리합니다.

핵심 원칙은 다음과 같습니다.

* 각 서비스는 자기 데이터만 소유한다.
* 다른 서비스의 DB에 직접 접근하지 않는다.
* 서비스 간 의존성은 API 또는 이벤트로만 연결한다.
* 영화 메타데이터와 사용자 컬렉션 데이터는 분리한다.
* 평점, 시청 상태, 소장 위치는 독립적으로 변경될 가능성이 있으므로 별도 책임으로 본다.

## 2. 사용자 유형

| 사용자     | 설명                               |
| ------- | -------------------------------- |
| 일반 사용자  | 자신의 영화 컬렉션을 등록, 조회, 수정, 삭제하는 사용자 |
| 관리자     | 공통 영화 메타데이터, 장르, 인물 정보를 관리하는 사용자 |
| 시스템 관리자 | 서비스 상태, 장애, 로그, 운영 설정을 관리하는 사용자  |

## 3. 기능 요구사항

### 영화 메타데이터 관리

* 영화 제목, 개봉 연도, 설명, 포스터, 러닝타임을 관리한다.
* 장르 정보를 관리한다.
* 배우, 감독 등 참여 인물 정보를 관리한다.
* 영화와 장르, 영화와 인물의 관계를 관리한다.

### 사용자 컬렉션 관리

* 사용자는 자신의 컬렉션에 영화를 추가할 수 있다.
* 사용자는 컬렉션에서 영화를 제거할 수 있다.
* 사용자는 보유 영화 목록을 조회할 수 있다.
* 컬렉션 항목에는 사용자 기준의 개인 메모를 추가할 수 있다.

### 시청 상태 관리

* 사용자는 영화별 시청 상태를 관리할 수 있다.
* 상태 예시는 다음과 같다.

  * 보고 싶음
  * 보는 중
  * 시청 완료
  * 보류
* 시청 완료일을 기록할 수 있다.

### 평점 및 리뷰 관리

* 사용자는 영화에 개인 평점을 부여할 수 있다.
* 사용자는 간단한 리뷰 또는 감상평을 작성할 수 있다.
* 사용자는 자신의 평점과 리뷰를 수정, 삭제할 수 있다.

### 소장 위치 관리

* 사용자는 영화의 소장 형태를 기록할 수 있다.

  * Blu-ray
  * DVD
  * 파일
  * 스트리밍 구매
  * 대여
* 사용자는 실제 보관 위치 또는 플랫폼명을 기록할 수 있다.

### 검색 및 조회

* 제목, 장르, 배우, 감독, 평점, 시청 상태, 소장 위치 기준으로 검색할 수 있다.
* 사용자별 컬렉션만 조회되어야 한다.
* 관리자 또는 공통 검색에서는 전체 영화 메타데이터 검색이 가능해야 한다.

## 4. 비기능 요구사항

| 구분      | 요구사항                                          |
| ------- | --------------------------------------------- |
| 확장성     | 영화 수, 사용자 수, 리뷰 수 증가에 따라 서비스별 독립 확장이 가능해야 한다. |
| 독립 배포   | 영화 메타데이터 서비스 변경이 평점 서비스 배포를 강제하지 않아야 한다.      |
| 데이터 소유권 | 각 서비스는 자기 DB를 소유하며, 타 서비스 DB에 직접 접근하지 않는다.    |
| 장애 격리   | 평점 서비스 장애가 컬렉션 조회 전체 장애로 확산되지 않아야 한다.         |
| 성능      | 컬렉션 목록 조회는 자주 사용되므로 캐싱 또는 조회 모델을 고려한다.        |
| 보안      | 사용자별 컬렉션, 평점, 리뷰는 본인만 수정 가능해야 한다.             |
| 관측성     | 서비스별 로그, 메트릭, 트레이싱이 가능해야 한다.                  |
| 일관성     | 서비스 간 데이터는 강한 일관성보다 최종적 일관성을 우선 고려한다.         |

## 7. Micro Service 후보

### 1) User Service

사용자 계정, 인증 주체, 사용자 역할을 관리합니다.

**책임**

* 사용자 정보 관리
* 사용자 식별자 발급
* 사용자 권한 정보 제공

**소유 데이터**

* User

**다른 서비스 의존성**

* 최소화
* 다른 서비스가 userId만 참조하도록 설계

---

### 2) Movie Catalog Service

공통 영화 정보를 관리합니다.

**책임**

* 영화 메타데이터 관리
* 장르 관리
* 배우/감독 정보 관리
* 영화 검색 제공

**소유 데이터**

* Movie
* Genre
* Person
* MovieGenre
* MovieCast
* MovieDirector

**다른 서비스 의존성**

* User Service에 의존하지 않음
* 사용자별 정보는 절대 포함하지 않음

---

### 3) Collection Service

사용자별 영화 컬렉션을 관리합니다.

**책임**

* 사용자의 영화 추가/삭제
* 내 컬렉션 목록 관리
* 컬렉션 메모 관리

**소유 데이터**

* CollectionItem

**다른 서비스 의존성**

* movieId만 참조
* Movie Catalog Service DB 직접 접근 금지
* 영화 상세 정보가 필요하면 API 조회 또는 읽기 모델 사용

---

### 4) Watch Status Service

사용자의 시청 상태를 관리합니다.

**책임**

* 보고 싶음, 보는 중, 시청 완료 등 상태 관리
* 시청 완료일 관리

**소유 데이터**

* WatchStatus

**다른 서비스 의존성**

* Collection Service에 강하게 의존하지 않는 것이 좋음
* userId, movieId 기준으로 독립 관리

---

### 5) Rating & Review Service

사용자의 평점과 리뷰를 관리합니다.

**책임**

* 평점 등록/수정/삭제
* 리뷰 등록/수정/삭제
* 영화별 평균 평점 계산 가능

**소유 데이터**

* RatingReview

**다른 서비스 의존성**

* Movie Catalog Service의 movieId만 참조
* 평균 평점은 이벤트 기반으로 별도 집계 가능

---

### 6) Ownership Service

사용자의 소장 형태와 위치를 관리합니다.

**책임**

* 물리/디지털 소장 형태 관리
* 보관 위치, 플랫폼명 관리

**소유 데이터**

* Ownership

**다른 서비스 의존성**

* Collection Service에 직접 종속시키지 않음
* userId, movieId 기준으로 독립 관리

---

### 7) Query / Aggregation Service

화면 조회에 필요한 데이터를 조합합니다.

**책임**

* 내 영화 컬렉션 통합 조회
* 영화 메타데이터, 시청 상태, 평점, 소장 위치 조합
* 프론트엔드에 최적화된 응답 제공

**소유 데이터**

* 선택 사항
* CQRS 적용 시 읽기 모델 소유 가능

**다른 서비스 의존성**

* 여러 서비스의 API 또는 이벤트 기반 Read Model 사용
* 핵심 도메인 데이터의 원천 소유자는 아님

## 8. 서비스별 책임 경계

| 서비스                     | 포함해야 하는 것      | 포함하면 안 되는 것       |
| ----------------------- | -------------- | ----------------- |
| User Service            | 사용자, 권한        | 영화, 컬렉션, 평점       |
| Movie Catalog Service   | 영화 공통 정보       | 사용자별 시청 상태, 개인 평점 |
| Collection Service      | 사용자의 컬렉션 포함 여부 | 영화 상세 메타데이터 관리    |
| Watch Status Service    | 사용자별 시청 상태     | 컬렉션 소유 여부 판단      |
| Rating & Review Service | 평점, 리뷰         | 영화 제목/장르 수정       |
| Ownership Service       | 소장 형태, 위치      | 영화 메타데이터          |
| Query Service           | 조회 조합          | 원천 데이터 변경 책임      |

## 9. 의존성 최소화 전략

가장 중요한 분리 기준은 **공통 영화 정보**와 **사용자 개인 데이터**를 분리하는 것입니다.

권장 의존 방향은 다음과 같습니다.

```text
Client
  ↓
API Gateway
  ↓
각 도메인 서비스

Movie Catalog Service ← 독립
Collection Service    → movieId만 참조
Watch Status Service  → userId, movieId만 참조
Rating Service        → userId, movieId만 참조
Ownership Service     → userId, movieId만 참조
Query Service         → 조회 전용 조합
```

피해야 할 구조는 다음과 같습니다.

```text
Collection Service → Movie Catalog DB 직접 조회
Rating Service → Collection Service에 영화 보유 여부 확인 강제
Watch Status Service → Collection Service 없이는 동작 불가
Ownership Service → CollectionItem 내부 필드로 종속
```

권장 방식은 다음과 같습니다.

```text
각 서비스는 userId, movieId를 참조 ID로만 사용한다.
상세 정보가 필요하면 API 또는 이벤트 기반 Read Model로 해결한다.
```

## 10. 추가 확인이 필요한 질문

* 사용자는 반드시 회원가입 후에만 컬렉션을 관리해야 하는가?
* 영화 메타데이터는 관리자가 직접 입력하는가, 외부 영화 API에서 가져오는가?
* 영화는 전역 공통 데이터인가, 사용자별 커스텀 영화 등록도 허용하는가?
* 하나의 영화에 여러 버전, 예를 들어 감독판/확장판을 구분해야 하는가?
* 평점은 개인 평점만 필요한가, 전체 평균 평점도 필요한가?
* 리뷰 공개 범위가 필요한가?
* 컬렉션 공유 기능이 필요한가?
* 가족/그룹 단위 컬렉션 관리가 필요한가?
* 소장 위치는 단순 텍스트인가, 별도 위치/플랫폼 마스터 관리가 필요한가?
* 검색은 단순 DB 검색으로 충분한가, 전문 검색 엔진이 필요한가?
* 서비스 간 데이터 정합성은 어느 수준까지 보장해야 하는가?

## 결론

초기 MSA 분해는 다음 구성이 적절합니다.

```text
User Service
Movie Catalog Service
Collection Service
Watch Status Service
Rating & Review Service
Ownership Service
Query / Aggregation Service
```

특히 **Movie Catalog Service**는 공통 영화 정보를, 나머지 서비스들은 **사용자별 행위와 상태**를 소유하도록 분리하는 것이 좋습니다. 이렇게 하면 영화 정보 변경, 평점 기능 변경, 시청 상태 기능 변경, 소장 위치 기능 변경이 서로를 강하게 끌고 가지 않게 됩니다.
